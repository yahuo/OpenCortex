# Lessons

- 2026-03-28: 在 agentic / multi-step retrieval 场景下，最终融合阶段必须复用前面步骤已经产出的检索结果，不能为了做最终排序再把 retriever 全量重跑一遍。实现多步检索时要显式检查“每一步调用了多少次 retriever”，把性能路径纳入回归测试，而不只验证功能正确性。
- 2026-04-07: 持久化会在查询时再按 scope（如 `kb`）过滤的数据结构时，不能先做全局截断再做 scope 过滤；截断策略必须优先保留同 scope 候选，并为“跨 scope 噪声耗尽预算”的最小复现场景补回归测试。
- 2026-04-07: 图边排序不能把“scope 优先级”放在“显式关系优先级”之前；显式链接/路径引用应先于启发式 same-series 边，否则会在无 scope 查询里丢掉真实跨文档引用。对多维排序规则要补成对回归测试，覆盖两个方向的截断场景。
- 2026-04-07: 图边截断排序里，“显式关系优先级”不仅要高于 scope 偏好，也要在显式关系内部保持强弱顺序；`links_to` 必须先于 `mentions_path`，不能因为同 KB 偏好让较弱的路径提及挤掉较强的跨文档显式链接。
- 2026-04-07: 有预算上限的扩展/搜索流程不能把 retriever 排序信号提前丢进 `set`；seed 去重必须保序，否则跨进程会因为哈希顺序不同而得到不稳定的候选范围。对这种逻辑要补“预算=1”之类的最小稳定性回归测试。
- 2026-04-07: 跨平台路径一旦参与索引键、manifest、symbol 或图边，就必须先统一成 canonical separator（这里用 `/`）；不要一边存宿主 OS 路径、一边用 `posixpath` 解析引用，否则 Windows 构建的索引会静默丢边。回归测试要覆盖“同 basename 多目录”的歧义场景。
- 2026-04-08: 设计文档的 Phase 1 不能同时引入“新能力接入运行时”和“大规模模块拆分”；首阶段必须优先做零风险、可独立验证的离线产物，先证明新增产物本身有价值，再决定是否接入在线检索链路。每个 Phase 都要写清楚量化验收标准，而不只写目标和改动项。
- 2026-04-08: 设计文档里的架构图如果展示的是目标态，必须在图前显式标注“非首阶段范围”，必要时补一个首个 PR 的简化图，避免读者把目标态误读成 Phase 1 施工清单。验收标准也不能用“部分”“一些”这类模糊词，必须绑定明确样本规模和成功阈值。
- 2026-04-08: 生成 Markdown 导航页时，显示文本和链接目标要分开处理；文本必须转义 Markdown 元字符，目标必须编码会破坏链接语义的字符（至少 `#`、空格、括号、`%`），否则本地文件明明存在，导航链接仍会失效。对这种问题要补特殊文件名回归测试，而不是只测普通路径。
- 2026-04-08: 生成 Markdown 预览页时，不能把原始预览文本直接塞进固定的三反引号围栏；如果预览本身含有 fenced code block，外层围栏必须动态选择另一种 marker 或更长的 fence，保证页面结构不被提前闭合。对这类问题要补“预览里自带 ```”的回归测试。
- 2026-04-08: 生成 Markdown 文件页时，路径派生 metadata 不能直接塞进固定的 inline code span；如果值里包含反引号，必须改用更长的 backtick fence 包裹，否则 metadata 区块会变成 malformed Markdown。对这种问题要补“文件名里含 `”的回归测试。
- 2026-04-09: 新增“回写到派生产物”的功能时，不能假设所有旧索引都已经具备新版本的派生文件；如果回写内容会链接到 `wiki/files/*` 之类的生成页，保存前必须显式校验这些派生产物存在，并在缺失时懒生成或拒绝写入。回归测试要覆盖“旧索引未重建但仍可运行”的场景。
- 2026-04-09: 面向 Docker/LAN 多用户场景的共享 Markdown/JSON 产物，不能用无保护的 read-modify-write 更新；像 `wiki/log.md` 这种共享日志必须加锁或使用原子合并写入，否则并发保存会静默丢记录，并在后续 lint 中表现为 orphan。对这类问题要补至少一个并发写入回归测试。
- 2026-04-09: 并发写协议不能只覆盖“用户动作”路径；凡是会重写同一共享产物的后台重建路径也必须参与同一把锁。否则即使 `save_query_note()` 自己串行了，`generate_wiki()` 这类 rebuild 仍会用旧快照覆盖新 log 条目。回归测试要直接验证“rebuild 在锁持有期间会等待，而不是偷跑重写”。
- 2026-04-09: 用 “创建锁文件 + 结束时 unlink” 模拟互斥时，必须处理 crash/kill 后残留锁文件；否则一次中断就会把后续所有写入永久楔死。优先使用内核管理、随进程退出自动释放的文件锁（如 POSIX `flock`）；无法使用时至少实现 stale-lock 检测/回收，并补“遗留锁文件仍可继续写”的回归测试。
- 2026-04-09: 不能把 Unix 上“无害”的进程探测习惯直接搬到 Windows；像 `os.kill(pid, 0)` 在 Windows 回退锁路径里可能走真正的 kill 语义。跨平台锁/进程探测要按平台分支实现，并补一条显式验证“Windows 分支不会调用 Unix 探测 API”的回归测试。
- 2026-04-09: 调用 Win32 API 的 `ctypes` 边界必须显式声明 `argtypes/restype`；默认 `c_int` 在 64 位 Windows 上会截断句柄，导致“偶发失效”这类高风险并发 bug。对这类 FFI 代码要补一条直接断言签名配置的回归测试，而不是只测上层业务逻辑。
- 2026-04-09: 读取派生产物的 UI 缓存键不能只绑主索引文件；如果用户动作会更新 `lint_report.json`、`wiki/` 或其他二级产物，而 `index.faiss` 不变，页面就会稳定展示旧摘要。对这类缓存要么把相关产物的 mtime 纳入 key，要么在写入成功后显式清 cache，最好两者都做。
